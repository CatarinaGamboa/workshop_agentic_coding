What Is This?

This repo contains experimental data from Barista, a system that uses AI agents (powered by Claude) to automatically generate typestate specifications for Java standard library classes.

Each of the 35 target classes was processed 5 independent times. Every run produced a specification (Java annotations), a state diagram (Mermaid), a state machine (JSON), and a detailed workflow log capturing cost, duration, tool calls, and agent turns.

The workshop goal is to parse, analyze, and write up findings from these 175 runs.

Key Concepts

Typestate
A protocol that tracks which methods are valid to call based on an object's current state. For example, you can't read() from a closed stream, or connect() an already-connected socket.
LiquidJava
A verification framework that checks Java code against typestate rules using annotations like @StateSet, @StateRefinement, and @Refinement.
Refinements File
A *Refinements.java file declaring the states and method contracts for a class — the main output of each Barista run.
Complexity Rating
Each class is rated none simple moderate or complex based on factors like branching transitions, scale, and encoding workarounds.

Output File Types

⚙
workflow_*.json — Agent execution log. Contains wall clock time, USD cost, SDK turns, tool call counts (Read, Write, Bash, MCP tools, etc.), agent spawn info, and per-agent breakdowns.
✍
*Refinements.java — The typestate specification. Java interface with LiquidJava annotations declaring valid states and method transitions (e.g., @StateRefinement(from="unconnected", to="connected")).
📈
*.mmd — Mermaid state diagram source. Shows states as nodes and method calls as labeled transitions. Can be rendered with any Mermaid viewer.
📄
*.json (in class dir) — Machine-readable state machine. Contains class metadata (name, signature, description, interfaces) plus constructor and method details.

Workflow Log Structure

Each workflow_*.json captures everything about one agent run:

FieldDescription
metrics.session.wall_clock_secsTotal wall clock time in seconds
metrics.session.total_cost_usdTotal API cost in USD
metrics.session.sdk_turnsNumber of SDK conversation turns
metrics.tool_callsCounts per tool: Read, Write, Edit, Bash, Grep, Glob, MCP tools, etc.
metrics.tool_errorsError counts per tool
metrics.agent_spawnsSub-agents invoked (protocol-analyzer, liquidjava-spec-writer, etc.)
metrics.agentsPer-agent breakdown of turns and tool calls
metrics.turnsOrdered list of every agent turn with tool call counts

Workshop Steps

  1. Explore — Understand the repo structure, data formats, and what typestate means
  2. Parse — Write a Python script to extract metrics from all 175 workflow logs into a CSV
  3. Analyze — Create a Jupyter notebook with statistics and visualizations (cost, duration, tool usage, complexity correlations)
  4. Review — Critically review the notebook for correctness, completeness, and clarity
  5. Write — Draft a results section in paper.tex with tables and figure references

Example: Socket Typestate

Here's a simplified view of the java.net.Socket state machine generated by Barista (from run 1):

States: Unconnected, Connected, InputShutdown, OutputShutdown, BothShutdown, Closed

[*] ──Socket()──> Unconnected
[*] ──Socket(host,port)──> Connected
Unconnected ──connect()──> Connected
Connected ──shutdownInput()──> InputShutdown
Connected ──shutdownOutput()──> OutputShutdown
InputShutdown ──shutdownOutput()──> BothShutdown
OutputShutdown ──shutdownInput()──> BothShutdown
any non-closed ──close()──> Closed
